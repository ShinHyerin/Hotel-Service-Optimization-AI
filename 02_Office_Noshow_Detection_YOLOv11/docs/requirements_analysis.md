# 📝 공유오피스 관제 및 노쇼 감지 시스템 요구사항 정의서

## 1. 시스템 개요 (System Overview)

본 시스템은 공유오피스 및 세미나실의 예약 데이터와 실시간 CCTV 카메라 피드(Vision)를 연동하는 B2B2C 공간 관리 플랫폼이다. 일반 고객에게는 최적의 예약 환경을 제공하고, 공유오피스 관리자(운영사)에게는 ML 기반 예약 취소율 예측 및 YOLOv11 기반 실시간 공간 이용 실태(노쇼/자리비움/미정리 퇴실) 관제 기능을 제공하여 공간 자원 운영 효율성을 극대화한다.

## 2. 기능적 요구사항 (Functional Requirements, FR)

### FR-01: 관리자 전용 Vision 관제 인터페이스 제공

- **FR-01-01:** 기존 Flask 웹 애플리케이션의 마스터 네비게이션 바에 **[Vision] 관제 메뉴**를 추가한다.
- **FR-01-02:** 역할 기반 권한 제어(RBAC)를 적용하여, 해당 메뉴는 **'관리자 계정'으로 로그인했을 때만 노출 및 접근**이 가능하도록 제한한다.
- **FR-01-03:** 관리자가 [Vision] 메뉴 클릭 시, 공유오피스 전체 룸의 실시간 이용 현황, 노쇼 위험 방 리스트, 실시간 탐지 피드가 포함된 대시보드 페이지로 이동한다.

### FR-02: 실시간 영상 분석 및 커스텀 멀티 클래스 탐지 (Multi-class Detection)

- **FR-02-01:** YOLOv11 모델을 활용하여 입력되는 영상 프레임에서 `Person`(사람), `Laptop`(노트북), `Cup`(커피잔/텀블러), `Paper`(문서/책/인쇄물)의 4대 핵심 객체를 실시간으로 동시에 탐지해야 한다.
- **FR-02-02:** 현장 고증 및 데이터 수집 환경을 반영하여 기존 `Bag`, `Phone` 클래스를 배제하고, 공유오피스 책상 위에서 가장 빈번하게 관찰되는 `Cup`과 `Paper`를 커스텀 클래스로 확정하여 탐지 정확도를 제고한다.

### FR-03: 멀티-클래스 조건부 인지 비즈니스 로직 적용

단순한 객체 카운팅을 넘어, 사람이 부재할 때(`Person` = 0) 남겨진 소지품 객체 간의 상관관계를 연산하여 세 가지 상태 코드를 관리자 대시보드에 실시간 매핑한다.

- **FR-03-01 (실제 노쇼 - No-Show):** 예약 시간이 되었으나 일정 시간 동안 `Person` = 0명이며, `Laptop`, `Cup`, `Paper` 등 모든 소지품 객체가 0개로 탐지될 경우 ➔ **[노쇼 위험]** 상태로 판별하여 경고를 발생시키고 자동 예약 취소 및 패널티 프로세스를 구동한다.
- **FR-03-02 (소지품 기반 자리비움 - Temporary Absence):** 자리에 사람은 없으나(`Person` = 0), 책상 위에 `Laptop`, `Cup`, `Paper` 중 하나 이상의 유효 객체가 탐지될 경우 ➔ **[잠시 자리비움]** 상태로 인지하여 억울한 노쇼 패널티 대상에서 예외 처리한다.
- **FR-03-03 (퇴실 후 미정리 - Dirty Vacancy):** `Person` = 0이고 `Laptop`도 없으나, 마시던 커피잔(`Cup`)이나 파지(`Paper`) 등 잔여 쓰레기만 탐지되는 경우 ➔ **[확인 필요: 미정리 퇴실]** 상태로 판별하여 관리자에게 청소 및 공간 정돈 알림을 송신한다.

### FR-04: RESTful API 기반 경량화 통신

- **FR-04-01:** 프론트엔드(CCTV 카메라 스트리밍 스크립트) 환경과 백엔드(YOLOv11 추론 엔진 및 로직 모듈) 간의 데이터 송수신은 JSON 포맷의 REST API를 통해 이뤄진다.
- **FR-04-02:** 전송 규격은 Base64로 인코딩된 이미지 프레임을 POST 요청으로 수신하고, 분석 결과(객체 좌표, 클래스, 판별 상태 코드)를 JSON 응답으로 반환한다.

## 3. 데이터 엔지니어링 및 학습 전략 요구사항 (Data Engineering)

### DE-01: 비디오 시퀀스 기반 데이터 수집 및 전처리

- **DE-01-01:** 단일 정적 이미지 데이터의 한계를 극복하고 실시간 CCTV 관제 환경의 변수(모션 블러, 각도, 조명 변화)에 대응하기 위해, 유튜브의 회의실 타임랩스 및 오피스 영상을 소스로 확보한다.
- **DE-01-02:** 다운로드한 비디오 스트림을 **초당 5~10 프레임(FPS)** 단위로 슬라이싱하여 연속성 있는 raw 데이터셋을 추출한다.
- **DE-01-03 (과적합 방지 샘플링):** 타임랩스 특성상 발생하는 동일 장면의 중복 데이터 양산을 막기 위해, Roboflow 업로드 시 프레임 레이트를 1 FPS 이하로 전처리 샘플링하여 징검다리 형태로 변화가 뚜렷한 이미지 컬렉션을 구축한다.

### DE-02: 클래스 불균형(Class Imbalance) 해소 및 데이터 밸런싱

- **DE-02-01:** 다수 클래스인 `Person`에 모델이 편향(Bias)되는 것을 차단하기 위해 `Person`, `Laptop`, `Cup`, `Paper` 인스턴스의 라벨링 타겟 비율을 **최대한 1:1:1:1 균등 분포**에 수렴하도록 통제한다.
- **DE-02-02:** 단일 프레임 내 소수 객체(`Cup`, `Paper`)가 동시에 다수 포함된 이미지 위주로 선별 주석 처리를 진행하며, 부족한 클래스는 데이터 증강(Data Augmentation - 회전, 밝기 변경, 크롭 등) 기법을 집중 적용하여 바운딩 박스 총개수의 균형을 강제한다.

### DE-03: 학습 및 추론 효율화를 위한 해상도 이원화 아키텍처 (🌟 핵심 역량)

- **DE-03-01 (학습 단계 - Train Phase):** 딥러닝 모델의 학습 속도 최적화, 경량화 및 손실 함수(Loss)의 고속 수렴을 유도하기 위해 원본 고화질 시퀀스를 다운샘플링(Downsampling)하여 **안정적인 고효율 저해상도 환경**에서 커스텀 모델 학습을 진행한다. (Roboflow 내 플랫폼 및 커널 기반 파이프라인 활용)
- **DE-03-02 (추론 단계 - Inference Phase):** 실제 Flask 서비스 운영 단계에서는 미세 객체(작은 컵, 얇은 종이 등)의 유실 및 미검출(False Negative)을 원천 방지하기 위해, 입력되는 원본 CCTV 비디오 스트림을 **고해상도 그대로 YOLOv11 엔진에 입력하여 객체 정밀도를 최고치로 확보**한다.

## 4. 비기능적 요구사항 (Non-Functional Requirements, NFR)

### NFR-01: 실시간성 및 성능 (Latency)

- **NFR-01-01:** 웹 인터페이스에서 REST API를 통해 이미지 분석을 요청하고, YOLOv11 추론 및 비즈니스 로직 연산을 거쳐 최종 UI 대시보드 화면에 반영되기까지의 **총 지연 시간(End-to-End Latency)은 1초 이내**여야 한다.
- **NFR-01-02:** 이를 위해 WSL2 가상환경 내에 `opencv-python-headless`를 빌드하여 디스플레이 버퍼 충돌을 방지하고 고속 추론 흐름을 상시 유지한다.

### NFR-02: 독립성 및 확장성 (Scalability)

- **NFR-02-01:** 기존 호텔 예약/취소율 프로젝트 코드와의 결합도를 낮추기 위해 **Flask Blueprint(블루프린트) 모듈화 설계**를 적용한다. 비전 관련 라우트(`vision_routes.py`)를 독립 관리하여 추후 세미나실 호수(Room ID) 확장 시 메인 서버 코드 수정 없이 엔드포인트를 유연하게 확장 가능하도록 조치한다.
